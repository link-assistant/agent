# Case Study: Issue #217 - Turn on `--summarize-session` by default with the same model

## Issue Reference
- Issue: https://github.com/link-assistant/agent/issues/217
- PR: https://github.com/link-assistant/agent/pull/218

## Timeline

- **2026-03-30**: Issue filed requesting `--summarize-session` be enabled by default, using the same model as `--model`, and improving context size management.

## Problem Statement

The `--summarize-session` flag was disabled by default (`false`), requiring users to explicitly enable it. When enabled, it used a "small model" (via `Provider.getSmallModel()`) for summarization rather than the model the user specified with `--model`. This caused inconsistent behavior.

Additionally, the compaction overflow detection had no safety margin - it triggered exactly when token count exceeded `context - output`, leaving no headroom for variance. Context size diagnostic data was also not exposed in JSON output, making it difficult to debug compaction decisions.

## Root Cause Analysis

### 1. `--summarize-session` default was `false`
- **Location**: `js/src/index.js` line 737, `js/src/flag/flag.ts` line 109
- **Why**: Originally disabled to save tokens and prevent rate limit issues with free tier models (issue #179)
- **Impact**: Users had to know about and explicitly enable this useful feature

### 2. Session summarization used a different model
- **Location**: `js/src/session/summary.ts` line 95
- **Code**: `Provider.getSmallModel(assistantMsg.providerID)` was used instead of the session's actual model
- **Impact**: Summaries generated by a different (smaller) model than the one the user chose, leading to quality inconsistencies

### 3. No safety margin in compaction overflow detection
- **Location**: `js/src/session/compaction.ts` lines 31-45
- **Code**: `isOverflow` checked `count > usable` with no margin
- **Impact**: Compaction triggered at the exact boundary, leaving no headroom for token counting variance

### 4. No context diagnostics in JSON output
- **Impact**: Impossible to debug compaction decisions from JSON output alone

## Solution

### Changes Made

1. **Enable `--summarize-session` by default** (`js/src/flag/flag.ts`, `js/src/index.js`)
   - Changed CLI default from `false` to `true`
   - Updated flag initialization to default to `true` unless explicitly disabled via env var
   - Users can disable with `--no-summarize-session` or `AGENT_SUMMARIZE_SESSION=false`

2. **Use the same model for summarization** (`js/src/session/summary.ts`)
   - Changed from `Provider.getSmallModel()` to `Provider.getModel(assistantMsg.providerID, assistantMsg.modelID)`
   - Uses the exact model from the session (matching `--model`)
   - Falls back gracefully with logging if model cannot be loaded

3. **Add 15% safety margin to compaction** (`js/src/session/compaction.ts`)
   - Added `OVERFLOW_SAFETY_MARGIN = 0.85` constant
   - Compaction triggers at 85% of usable context (15% headroom)
   - Added detailed logging of overflow check parameters

4. **Add context diagnostics to JSON output** (`js/src/session/message-v2.ts`, `js/src/session/processor.ts`)
   - New `ContextDiagnostics` schema with fields: `contextLimit`, `outputLimit`, `usableContext`, `safeLimit`, `safetyMargin`, `currentTokens`, `headroom`, `overflow`
   - Added as optional `context` field on `StepFinishPart`
   - Computed via `SessionCompaction.contextDiagnostics()` helper

### Compaction Math Example (minimax-m2.5-free)
- Context limit: 204,800 tokens
- Output limit: min(131,072, 32,000) = 32,000 tokens
- Usable context: 204,800 - 32,000 = 172,800 tokens
- Safe limit (85%): floor(172,800 * 0.85) = 146,880 tokens
- Headroom before compaction: ~25,920 tokens (15% margin)

## Files Changed

- `js/src/flag/flag.ts` - Default `SUMMARIZE_SESSION` to true
- `js/src/index.js` - CLI default and middleware for `--summarize-session`
- `js/src/session/summary.ts` - Use same model instead of small model
- `js/src/session/compaction.ts` - Safety margin + diagnostics helper + overflow logging
- `js/src/session/message-v2.ts` - ContextDiagnostics schema on StepFinishPart
- `js/src/session/processor.ts` - Populate context diagnostics in step-finish parts
